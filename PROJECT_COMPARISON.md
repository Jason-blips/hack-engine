# 项目对比：demo vs githack-app

## 一、项目概览

| 维度 | **demo**（根项目） | **githack-app** |
|------|-------------------|------------------|
| **位置** | 项目根目录 | `demo/githack-app/` |
| **类型** | 后端 + 服务端渲染（SSR） | 纯前端 SPA（React） |
| **技术栈** | Spring Boot 3、Spring Security、JPA、MySQL、Thymeleaf | React 18、React Router、styled-components、axios |
| **认证方式** | 真实：BCrypt 存库 + Session | 仅 localStorage 布尔值，无后端 |
| **数据库** | MySQL（生产）/ H2（测试） | 无 |

---

## 二、功能对比

### demo（登录系统模板）

- **注册**：用户名 + 密码，BCrypt 加密存库，重复用户名校验。
- **登录**：表单提交到 Spring Security，校验数据库用户，Session 保持。
- **登出**：清除 Session，跳转登录页。
- **访问控制**：未登录访问 `/welcome` 会重定向到 `/login`。
- **测试**：MockMvc + H2 自动化测试，无需 MySQL。

### githack-app（前端演示）

- **登录**：任意输入 → 约 1.5 秒后直接 `onLoginSuccess()`，无后端校验。
- **注册**：多步（手机验证码 → 邮箱验证码 → 滑块 → 设密码），手机验证码打控制台，邮箱/注册 API 指向占位 URL。
- **验证码页**：写死验证码 `040322`（见 `constants.js`）。
- **支付密码页**：写死支付密码 `040322`，通过后跳转支付成功页。
- **支付成功页**：展示订单信息（前端构造）。
- **“登录态”**：`localStorage` 的 `githack_logged_in`，无服务端参与。

---

## 三、相似与差异

### 相似点

- 都有「登录 / 注册」的页面和流程。
- 都有「未登录则跳转到登录页」的受保护路由（demo 用 Spring Security，githack 用 `ProtectedRoute` + `isLoggedIn`）。

### 核心差异

| 方面 | demo | githack-app |
|------|------|-------------|
| **认证是否真实** | ✅ 是 | ❌ 否（前端模拟） |
| **是否适合当“登录系统”** | ✅ 是 | ❌ 否（仅 UI/流程演示） |
| **是否有后端** | ✅ 有（Java） | ❌ 无 |
| **UI 风格** | 简洁、深色、模板级 | 渐变、多步骤、组件更丰富 |
| **额外业务** | 无 | 验证码、支付密码、支付成功演示 |

---

## 四、综合结论：哪个更好？

- **若目标是一个「可用的登录系统模板」或「后端认证学习项目」**  
  → **demo 更好**：有真实认证、数据库、测试，可直接当新项目起点。

- **若目标是一个「纯前端演示 / Hackathon 展示 / 不接后端的作业」**  
  → **githack-app 更合适**：UI 完整、流程多，但**不能**当作真实登录系统使用。

两者**定位不同**：demo 是「可落地的登录后端+简单前端」，githack-app 是「无后端的登录+验证码+支付流程前端演示」。

## 五、取舍与合并建议

### 方案 A：只保留一个

- **只做登录系统 / 后端学习** → **保留 demo，删除或移出 githack-app**。  
  githack-app 没有真实认证，和“登录系统”目标不一致。
- **只做前端演示、不接后端** → 可只保留 githack-app，但要清楚：这不是“登录系统”，只是前端流程演示。

### 方案 B：合并成一个项目（前后端分离）

把 **demo 当作后端**，**githack-app 当作前端**，合并成一个仓库里的「一个产品」：

1. **后端（保持 demo）**  
   - 已有：注册、登录、Session、BCrypt。  
   - 可选：增加 REST API（如 `/api/login`、`/api/register`、`/api/me`），或继续用表单登录 + Cookie Session，前端通过代理访问。

2. **前端（改造 githack-app）**  
   - 登录：改为调用后端登录接口（或表单 POST 到后端），不再“任意输入就通过”。  
   - 注册：可简化为“用户名+密码”调 demo 的注册 API；若保留多步注册，需后端提供验证码/邮箱等接口。  
   - 验证码 / 支付密码：要么去掉写死的 `040322`，要么改为后端下发/校验（否则合并后仍是演示逻辑）。

3. **部署与开发**  
   - 开发：前端 `npm run start`（可 proxy 到 demo 的 8080），后端 `./mvnw spring-boot:run`。  
   - 合并后目录示例：根目录保留 `pom.xml`、`src`（后端），`githack-app` 或 `frontend` 为前端。

**合并的工作量**：中等偏上（后端加 API、前端改认证与请求、可能加 CORS/代理）。**收益**：一个仓库内既有可用的登录后端，又有现代前端 UI，适合当「前后端分离登录 + 扩展业务」的模板。

## 六、直接回答你的问题

- **两个项目是否相似？**  
  表面相似（都有登录/注册页、保护路由），但**本质不同**：一个是真实登录后端+简单前端，一个是纯前端演示（无后端、验证码/支付写死）。

- **哪个更好？**  
  - 做**登录系统** → **demo 更好**。  
  - 做**前端展示/不接后端** → **githack-app 更合适**。

- **取舍建议**  
  - 只留一个：以「登录系统」为目标就**留 demo**，可删除或移出 githack-app。  
  - 合并：以「一个完整前后端模板」为目标，则 **demo 做后端 + githack-app 做前端**，并让前端真正调后端登录/注册，去掉写死验证码/支付密码（或改为后端实现）。
